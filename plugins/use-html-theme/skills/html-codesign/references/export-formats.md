# Export formats — codesign-answers documents

Exports are **their own document kind** (`codesign-answers`), deliberately
NOT the input spec: the input is the question set, the output is the
decision. Both formats (MD for humans, JSON for machines) export from both
control bars, in the variant selected by the bar's Slim/Full toggle.

**Slim is the default.** A paste-back to the agent should stay slim — the
agent authored the page and already holds the contexts, so re-sending them
is tokens spent on nothing. Full exists for human decision records (PRs,
Slack, `docs/decisions/`) where the reasoning must travel with the
decision.

## Slim JSON (default)

```json
{
  "kind": "codesign-answers",
  "version": "1",
  "variant": "slim",
  "title": "Dashboard redesign — pick a direction",
  "exported_at": "2026-07-16T09:47:21-07:00",
  "answers": [
    {
      "section": "sec-01-direction",
      "question": "Overall direction",
      "selected": [ { "id": "ch-01-a", "label": "Focused — one hero number + one chart" } ],
      "note": "Focused matches the five-second status check."
    }
  ],
  "feedback": "Can we see a hybrid before committing?"
}
```

- `note` and `feedback` appear only when non-empty; `selected` is `[]` for
  an unanswered section.
- A **skipped** section keeps its entry but replaces `selected` with
  `"skipped": true` — the loop can tell *deliberately not decided* from
  *never asked* and won't blindly re-ask.
- An **open question** (typed via "Ask a question") appears as
  `"open_question": "<text>"` on the entry — `"question"` is already the
  section title. Skip and open_question can coexist.
- Collapse/hide/ask-visibility state never appears — identical picks export
  identically regardless of how the page was folded.

## Full JSON (toggle)

Each answer additionally carries:

```json
"context": { "summary": "…", "recommendation": "…", "recommended": ["ch-01-a"] },
"options": [ { "id": "ch-01-a", "label": "…", "selected": true },
             { "id": "ch-01-b", "label": "…", "selected": false } ],
"verdict": "followed"
```

`context` is the **envelope only** — the free-form body (charts, tables,
images) is not extracted; the self-contained page file is the durable
rich-context archive. `options` is omitted on skipped sections. `verdict`
∈ `followed` · `went-against` · `unanswered` · `skipped` ·
`no-recommendation` — derived by comparing `selected` against
`recommended` (exclusive: the pick is in `recommended`; pick-any: the
selected set equals the recommended set).

## Slim Markdown (default)

The decision, nothing else — ID, question, chosen answer(s), note. Skipped
sections are **omitted entirely** (unless they carry an open question, in
which case only the question line renders — questions always surface; skip
governs decision content only):

```markdown
# Dashboard redesign — pick a direction

> _Choices made: 4 of 9 · 1 skipped · Exported 2026-07-18 · slim export_

## sec-01-direction · Which direction should the dashboard take?

- ✓ `ch-01-a` Focused — one hero number + one chart

> ❓ **Open question (`q-01`):** does mobile get the same layout?

> **Note (`note-01`):** Focused matches the five-second status check.

## sec-02-db · Which database for v1?

- (no selection)

_Generated by html-codesign · slim export · 2026-07-18 · 3 sections · 9 choices · 4 selected · 1 skipped._
```

(sec-03 was skipped with no question — it simply isn't in the record.)

## Full Markdown (toggle)

Reads like a lightweight ADR — context envelope, recommendation, all
options, verdict, notes:

```markdown
## sec-02-db · Which database for v1?

**Context:** Zero-ops persistence vs a managed service. _(full context,
incl. any charts, lives in the page file)_

★ **Recommended:** `ch-02-a` — mature migrations story.

- `ch-02-a` ✗ Postgres
- **`ch-02-c` ✓ SQLite** ⚠ _went against recommendation (ch-02-a)_
- `ch-02-d` ✗ Managed Firestore

> **Note (`note-02`):** want zero-ops for v1
```

Rendering rules: section headers are `## {section id} · {title}` (ID first
so it is unmissable when quoted in chat); the context line is the
envelope's `summary` plus the page-file pointer — the free-form body is
never text-extracted (lossy and nondeterministic); selected lines are
bolded whole; the verdict rides the selected line in exclusive sections
and follows the option list as its own line in pick-any sections; open
questions render as `❓ **Open question (`q-NN`):**` blockquotes; skipped
sections are omitted (question-only line if one is open); empty notes are
omitted; footer is an italic provenance line naming the skill, variant,
export date, and counts (including skips).

## Filenames

```
{title-slug}-{ISO-date}.{slim|full}.{md|json}
dashboard-redesign-2026-07-16.slim.json
```

The page suggests the slugged name in the export panel title; the browser
can't see existing files, so if you save a same-day re-export, append a
counter yourself: `dashboard-redesign-2026-07-16-2.slim.json`.

## Validation

`validate_spec.py` validates inputs only and rejects `codesign-answers`
documents with a pointer here. Answers documents are machine-produced by
the page's engine; the SKILL.md smoke test covers their shape.
