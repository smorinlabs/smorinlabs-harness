Heads up. The billing webhook payload now nests line_items under a billing object. If you parse webhooks directly (rather than going through the SDK), update before upgrading — see "Breaking changes" below.

What's new

New

Multi-region failover

The platform now serves traffic from us-east-1 and us-west-2 simultaneously, with sub-second failover when one region degrades. Health checks run every 5 seconds; switchover is transparent to clients.

New

Bulk edit for ops

Select multiple records in any list view, then apply the same change across all of them. Supports up to 500 records per operation. The bulk log preserves diffs so the operation can be undone within 24 hours.

Improved

Sync API p95 down 38%

Connection pool size raised from 100 to 250 per worker; idle connections now reaped after 30s instead of 5 minutes. The benefit is most visible on bursty workloads — under steady load the win is smaller (around 12%).

Fixed

Webhook retries no longer double-fire on 5xx

A race condition between the primary retry path and the dead-letter queue could result in two deliveries of the same webhook. Fixed; if your endpoint had defensive idempotency keys, you can keep them — they don't hurt.

Breaking changes

One change requires action before you upgrade.

Breaking

Webhook payload structure for billing events

The line_items array is now nested under a new billing object. Before:

v2.13 webhook payload
{
  "event": "invoice.paid",
  "line_items": [...]
}

After:

v2.14 webhook payload
{
  "event": "invoice.paid",
  "billing": {
    "line_items": [...],
    "subtotal_cents": 12000,
    "tax_cents": 990
  }
}

If you use the official SDKs (v2.14+), nothing to do — the SDK abstracts this. If you parse webhooks by hand, update your parser before upgrading.

Migration checklist

  • Search your codebase for line_items in webhook handlers.
  • Update the path from payload.line_items to payload.billing.line_items.
  • If you depend on subtotal or tax, they're now exposed on the billing object — previously you had to compute them.
  • Test against the staging environment using a sample webhook from the dashboard before flipping production.
